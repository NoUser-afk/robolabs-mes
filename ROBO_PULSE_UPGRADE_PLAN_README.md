# RoboPulse MES - общий план доработки

Дата составления: 2026-06-11  
Проект: `robolabs-mes-demo`  
Назначение документа: единый рабочий план, по которому можно последовательно улучшать RoboPulse MES без потери контекста между итерациями.

## 1. Как использовать этот файл

Этот документ является главным чеклистом для следующих доработок. При начале новой итерации нужно:

1. Прочитать разделы `2-5`, чтобы понимать цель, текущие ограничения и правила работы.
2. Взять первый незавершенный блок из раздела `7. Дорожная карта`.
3. Перед изменениями проверить связанные файлы и текущий UI.
4. После изменений выполнить критерии приемки из соответствующего блока.
5. Обновить чекбоксы в этом файле только после фактической проверки.

Статусы задач:

- `[ ]` - не начато.
- `[~]` - в работе, есть частичная реализация или требуется проверка.
- `[x]` - реализовано и проверено.
- `[!]` - заблокировано, нужна отдельная договоренность или внешняя информация.

Правило качества: не считать задачу готовой только потому, что код написан. Задача готова, когда UI/endpoint реально проверены, сборка проходит, а сценарий приемки воспроизводится.

## 2. Цель развития системы

RoboPulse должен перейти из состояния насыщенного demo/MVP в более устойчивую производственную систему для диспетчеризации заказов, запуска партий и поштучного прохождения операций по участкам.

Целевая система должна:

- давать диспетчеру понятный план производства, запуск партий и контроль очередей;
- давать работнику терминал только своего участка с минимальным количеством лишних действий;
- давать технологу удобный конструктор техпроцесса с редактированием, связями и проверками;
- давать директору мониторинг состояния производства, узких мест и рисков;
- сохранять историю действий: кто, когда и что начал, поставил на паузу, завершил или изменил;
- защищать производственные действия ролями и авторизацией;
- быть достаточно разложенной по модулям, чтобы новые требования не ломали существующие сценарии.

## 3. Текущее состояние по аудиту

Сильные стороны:

- Есть рабочий React/Vite frontend, NestJS backend, PostgreSQL/Prisma и Docker Compose.
- Есть роли: терминал, диспетчер, технолог, директор, администратор.
- Есть QR-вход терминала, HTTPS для мобильной камеры, PIN-вход терминала.
- Есть пульт диспетчера, терминал участка, директорский монитор, номенклатура, техпроцесс, граф процесса, партии, импорт Excel.
- Есть unit-level модель производства: `ProductionRun`, `ProductionUnit`, `ProductionUnitOperation`, `ProductionOperationEvent`.
- Терминал участка уже ориентирован на текущую операцию, очередь, старт/завершение и групповые действия только для разрешенных типов.
- В UI появилась идентичность RoboPulse: pulse-интро, анимации входа, старта и завершения операции.

Главные проблемы:

- Серверная авторизация неполная: многие производственные endpoints доступны без общего guard и проверки роли.
- Frontend почти целиком находится в `frontend/src/main.tsx`; файл слишком большой для дальнейшего безопасного развития.
- Backend-бизнес-логика сосредоточена в `backend/src/mes.service.ts`; сервис делает слишком много.
- Производственные данные частично нормализованы в Prisma, но сохраняется legacy-совместимость через `ProductionRunRecord.data Json`.
- Навигация по ролям хранит последнюю вкладку слишком грубо и может открывать неестественный стартовый экран.
- Конструктор техпроцесса функционален, но требует упрощения навигации по большим маршрутам.
- Обратная связь часто сделана через `alert(...)`, что не подходит для промышленного интерфейса.
- Нет полноценного e2e smoke-набора для ключевых ролей.

## 4. Обязательные правила выполнения доработок

### 4.1. Не ломать данные

- Не выполнять `docker compose down -v`, если пользователь явно не попросил удалить данные.
- Не удалять production/runtime volume без отдельного подтверждения.
- Миграции БД делать только через Prisma migrations или явно описанный migration script.
- Legacy JSON/record слои удалять только после миграции и проверки обратной совместимости.

### 4.2. Не расширять монолит без причины

Если задача затрагивает frontend, сначала оценить, можно ли вынести код в отдельный компонент/hook/API-модуль. Новые крупные экраны не добавлять в конец `main.tsx` как еще один большой блок.

Если задача затрагивает backend, не добавлять новый большой workflow в `MesService`, если его можно вынести в отдельный сервис.

### 4.3. Проверять UI, а не только сборку

Для пользовательских изменений обязательно проверять хотя бы один реальный сценарий в браузере:

- вход/выход;
- открытие нужной роли;
- целевое действие;
- отсутствие визуальных поломок в первом viewport;
- отсутствие релевантных console errors.

### 4.4. Сохранять производственную простоту

Интерфейс для рабочих терминалов должен быть проще, чем интерфейс диспетчера и технолога. На терминале не должно быть лишних справочников, сложных графов и действий, которые не относятся к текущему участку.

## 11. Риски и решения

### Риск: система остается demo, но используется как production

Решение: P0-задачи по авторизации и ролям выполнять до крупных новых возможностей.

### Риск: правки конструктора ломают производственный workflow

Решение: валидировать техпроцесс до сохранения и добавить smoke-сценарий запуска по измененному маршруту.

### Риск: bulk-action завершает не те операции

Решение: backend должен сам проверять совместимость selected operations, а frontend только помогать пользователю.

### Риск: JSON/legacy слой вызывает долгие транзакции

Решение: перейти к адресным updates в Prisma и оставить JSON только как временный compatibility/backup слой.

### Риск: UI красиво выглядит, но оператору неудобно

Решение: терминал держать максимально утилитарным: текущая операция, причина блокировки, крупные действия, минимум отвлекающих элементов.

## 12. Приоритет следующей итерации

Если продолжать прямо с этого места, рекомендуемый порядок:

1. P0 - общий AuthGuard/RoleGuard на backend.
2. P1 - role-scoped navigation и стартовые экраны по ролям.
3. P1 - сообщения QR/камеры и группировка терминалов на входе.
4. P1 - поиск/минимап/inspector в конструкторе техпроцесса.
5. P1 - причины блокировок и история действий на терминале.
6. P2 - адресные updates production operations вместо массовой перезаписи runs.
7. P2 - разбиение `main.tsx` и `MesService`.

Главный принцип: сначала защищаем производственные действия и делаем поведение предсказуемым, затем улучшаем удобство, затем чистим архитектуру.
