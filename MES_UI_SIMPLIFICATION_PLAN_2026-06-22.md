# План доработки интерфейса MES: меньше анимаций и служебных полей

Дата: 22.06.2026  
Стенд проверки: `http://localhost:8088/`  
Основание: обратная связь руководителя и работников от 19.06.2026, самостоятельный просмотр интерфейса после актуализации локального сайта с `ttm-mini`.

## 1. Цель доработки

Сделать интерфейс MES спокойнее, понятнее и короче для производственных ролей:

- убрать лишние пульсации, сканирующие линии и вступительные анимации;
- снизить визуальный шум на рабочих экранах;
- скрыть внутренние технические идентификаторы там, где они не помогают пользователю;
- сделать номер заказа и название изделия основными ориентирами;
- оставить служебные поля только в местах диагностики, администрирования или карточке с расширенными сведениями;
- подготовить интерфейс к последующим группировкам и фильтрам по понятным названиям.

## 2. Что считаем служебной информацией

Кандидаты на скрытие из основных пользовательских экранов:

- внутренние запуски вида `RUN-178...`;
- коды операций вида `ОР-00027`, если рядом уже есть название операции;
- `Unit` как пользовательская метка;
- логины терминалов вида `terminal.23`;
- технические подписи `Production runs`;
- поля `confidence`, `sourceRow`, `Op.ID`;
- внутренние связи маршрута `←`, `→`, `пред`, `след`, если это не экран редактирования техпроцесса;
- уровни маршрута `ур.`, `ряд`, `№ sequence`, если это не режим технолога;
- машинные коды причин простоя `NO_MATERIAL`, `WAIT_PREVIOUS_STAGE` без человеко-понятного названия.

Служебная информация может оставаться:

- в карточке расширенной диагностики;
- в режиме администратора или технолога;
- в tooltip / раскрываемом блоке "Технические данные";
- в URL, API и внутренних структурах данных;
- в логах, тестах и backend-моделях.

## 3. Основные принципы интерфейса

1. На первом уровне показывать только то, по чему человек принимает решение.
2. Главные поля: номер заказа, изделие, участок, операция, статус, готовность, действие.
3. Вторичные поля: срок, приоритет, количество, исполнитель, причина блокировки.
4. Технические ID не должны конкурировать с номером заказа.
5. Любая таблица должна иметь понятные названия колонок без сокращений, если место позволяет.
6. Анимация допустима только как короткая обратная связь на действие, без бесконечных пульсаций.
7. На мобильной ширине приоритет выше у навигации, заказа, изделия и действия; служебные детали скрывать раньше всего.

## 4. Зафиксированные проблемы по экранам

### 4.1. Вход и выбор терминала

Проблемы:

- при выборе участка видны логины `terminal.xx`;
- есть декоративная пульсация логотипа и фоновые эффекты;
- работнику не нужен технический логин терминала.

План:

- заменить `terminal.xx` на нейтральную подпись: "Терминал участка" или убрать строку полностью;
- оставить логин только в служебном раскрытии для администратора;
- отключить бесконечную пульсацию логотипа и фона.

Критерии приемки:

- на экране выбора терминала не видно `terminal.`;
- участки выбираются по названию;
- нет постоянных пульсаций после загрузки.

### 4.2. План производства

Проблемы:

- номер заказа визуально конкурирует с `RUN-*`, `ОР-*`, зависимостями и служебными колонками;
- в таблице есть колонки `Запуск`, `Ед.`, `Завис.`, которые понятны не всем ролям;
- в блокировках показываются коды операций вместо названий;
- текущее описание "Сгруппировано по номенклатуре..." выглядит как служебное пояснение.

План:

- сделать номер заказа главным текстом строки или карточки;
- убрать `RUN-*` из основной таблицы;
- заменить `Запуск` на "Партия" или скрыть из основного вида;
- заменить `Ед.` на "Единица" или показывать как `3 из 10`;
- заменить `Завис.` на понятное состояние: "Ожидает предыдущую операцию";
- в блокировках показывать название операции и участок, а не только `ОР-*`;
- перенести технический ID запуска в раскрываемую строку "Технические данные";
- добавить визуальное разделение: заказ -> изделие -> участок/операция -> действие.

Критерии приемки:

- номер заказа читается первым в строке;
- на первом экране плана не видно `RUN-*`;
- `ОР-*` не отображается без названия операции;
- причины ожидания читаются как фраза, а не как код;
- таблица остается компактной и не требует горизонтального скролла на обычном desktop.

### 4.3. Терминал участка

Проблемы:

- в карточках и истории видна метка `Unit`;
- операции показываются как `ОР-00035 Название`;
- работнику важнее название операции, изделие, заказ и действие;
- текущие анимации терминала отвлекают: сканирующие линии, пульсации, подсветки.

План:

- заменить `Unit` на "Единица" или убрать бейдж, если он не несет смысла;
- показывать операцию как "Балансировка вентилятора", а код `ОР-*` перенести в раскрытие;
- в истории действий показывать: операция, заказ, исполнитель, время, действие;
- убрать бесконечные scan/pulse эффекты в шапке, текущей операции и progress bar;
- оставить короткую смену состояния кнопки после действия, если она реально помогает.

Критерии приемки:

- работник видит название операции без необходимости расшифровывать код;
- `Unit` отсутствует;
- история действий не содержит внутренних run ID при наличии номера заказа;
- нет постоянной анимации в рабочем состоянии терминала.

### 4.4. Директор

Проблемы:

- KPI `Production runs` на английском;
- риск-блоки выводят `ОР-*` как заголовок;
- в списках решений встречаются внутренние run ID;
- на управленческом экране слишком много производственно-технических деталей.

План:

- заменить `Production runs` на "Запущенные партии" или "Партии в производстве";
- заголовок риска строить по названию операции и участку;
- номер заказа показывать раньше операции;
- внутренний run ID скрывать, если есть номер заказа или партия;
- оставить `ОР-*` только в раскрытии "Технические данные".

Критерии приемки:

- все KPI подписаны на русском и понятны руководителю;
- риск можно понять без знания кодов операций;
- управленческий экран отвечает на вопросы: где проблема, какой заказ, что мешает, кто должен действовать.

### 4.5. Граф процесса

Проблемы:

- много кодов `ОР-*`, `unit`, `Ожидает: ОР-*`;
- граф полезен технологу, но перегружает обычного пользователя;
- связи и уровни показываются как техническая модель.

План:

- разделить режимы: "Рабочий вид" и "Технический вид";
- в рабочем виде показывать название операции, участок, статус и блокировку;
- коды операций, уровни и связи оставить в техническом виде;
- заменить `Ожидает: ОР-*` на "Ожидает завершения: Название операции".

Критерии приемки:

- рабочий вид графа понятен без кодов операций;
- технологический режим сохраняет возможность диагностики связей;
- термин `unit` не отображается пользователю.

### 4.6. Номенклатура и техпроцесс

Проблемы:

- видны `Op.ID`, `confidence`, `sourceRow`, `Деталь / узел`, связи `←` и `→`;
- блоки маршрута показывают наборы `ОР-*`;
- для просмотра номенклатуры это избыточно.

План:

- для карточки номенклатуры сделать компактный маршрут: участок -> операция -> норма;
- `confidence` и `sourceRow` убрать из интерфейса просмотра;
- `Op.ID` заменить на "Код" только в режиме технолога;
- `Деталь / узел` переименовать в более понятное поле после уточнения бизнес-смысла;
- списки `ОР-*` в блоках заменить на количество операций и названия ключевых этапов.

Критерии приемки:

- в режиме просмотра номенклатуры нет `confidence`, `sourceRow`, `Op.ID`;
- маршрут читается как производственная последовательность;
- технический редактор техпроцесса не теряет нужные поля.

### 4.7. Справочники

Проблемы:

- в справочнике участков видны `terminal.xx`;
- в справочнике операций основным полем идет код `ОР-*`;
- часть таблиц воспринимается как технические реестры, а не справочники для пользователя.

План:

- в справочнике участков показывать название участка, статус, назначенный терминал человеко-понятно;
- логин `terminal.xx` показывать только в отдельной колонке "Логин" для администратора или в раскрытии;
- в справочнике операций первым показывать название операции, затем участок и норму;
- код операции сделать вторичным.

Критерии приемки:

- справочники можно просматривать по названиям;
- технические логины не видны обычному диспетчеру без необходимости;
- коды операций не являются главным визуальным якорем.

### 4.8. Смены и KPI

Проблемы:

- причины простоев могут показывать машинные коды вроде `NO_MATERIAL`;
- категории выглядят технически;
- руководителю и мастеру нужны названия причин и итоговое влияние.

План:

- показывать человеко-понятное название причины как основное;
- машинный код скрыть или перенести в техническое раскрытие;
- категории перевести на русские названия: материал, документация, ожидание, техническая причина;
- добавить краткий итог: сколько часов потеряно и по какой основной причине.

Критерии приемки:

- на экране KPI нет необъясненных машинных кодов;
- причины можно использовать в разговоре с производством без расшифровки.

### 4.9. Партии и архив

Проблемы:

- в списках партий и архиве видны `RUN-*`;
- карточка запуска содержит внутренний ID как заметное поле;
- в архиве колонка "Запуск" показывает технический идентификатор.

План:

- основным идентификатором сделать номер заказа или номер партии;
- `RUN-*` перенести в раскрытие "Технический ID";
- в архиве заменить колонку "Запуск" на "Партия / заказ";
- в карточке запуска показывать технический ID только в нижнем блоке.

Критерии приемки:

- список партий можно читать по заказам и изделиям;
- `RUN-*` не виден в основной таблице архива;
- при необходимости администратор может найти технический ID.

## 6. План сокращения служебных полей

### 6.1. Ввести единый helper отображения

Нужен набор функций форматирования, чтобы не прятать поля точечно в каждом компоненте:

- `displayOrderNumber(run)` -> номер заказа или "ручной запуск";
- `displayBatchNumber(run)` -> номер партии или короткое человеко-понятное имя;
- `displayOperationTitle(operation)` -> название операции без обязательного `ОР-*`;
- `displayOperationDetails(operation)` -> участок, статус, причина ожидания;
- `displayTechnicalId(value)` -> только для раскрытия/админ-режима;
- `displayReason(reason)` -> человеко-понятное название причины простоя.

Возможные места:

- `frontend/src/components/common.tsx`
- отдельный файл `frontend/src/domain/display-labels.ts`

### 6.2. Правило вывода ID

Пользовательский вид:

- показываем `orderNumber`, `batchNumber`, `productName`, `section`, `operation.name`;
- не показываем `run.id`, `unitId`, `operationId`, `terminal.login`.

Технический вид:

- показываем внутренние ID в раскрытии;
- используем мелкий нейтральный стиль;
- не ставим ID в заголовки и основные колонки.

### 6.3. Роли

Предложение по видимости:

| Роль | Основной вид | Технические ID |
| --- | --- | --- |
| Работник терминала | Название операции, заказ, изделие, действие | Скрыты |
| Диспетчер | Заказ, партия, участок, статус, блокировка | В раскрытии |
| Директор | Заказ, проблема, участок, влияние | Скрыты |
| Технолог | Может видеть коды операций и связи | Видны в техрежиме |
| Администратор | Может видеть логины и ID | Видны в справочниках/раскрытии |

## 9. Проверка после реализации

### 9.1. Автоматическая проверка DOM

Проверить, что на основных пользовательских экранах нет:

- `RUN-`;
- `terminal.`;
- `Production runs`;
- `Unit`;
- `confidence`;
- `sourceRow`;
- `NO_MATERIAL`;
- `WAIT_PREVIOUS_STAGE`.

Для `ОР-*` правило мягче: код может быть в техрежиме, но не должен быть главным заголовком в рабочих экранах.

### 9.2. Визуальная проверка

Обязательные экраны:

- План производства, desktop;
- План производства, mobile 390px;
- Терминал участка, desktop;
- Терминал участка, mobile 390px;
- Директор;
- Партии;
- Архив;
- Номенклатура.

Проверить:

- нет постоянных пульсаций;
- нет наложения текста;
- номер заказа виден первым или одним из первых;
- основные действия доступны без прокрутки там, где это возможно;
- технические ID не бросаются в глаза.

### 9.3. Функциональная проверка

- вход диспетчера;
- выбор терминала;
- фильтр плана;
- переключение вкладок;
- открытие партии;
