# Доработки RoboPulse по итогам совещания 10.06.2026

## 1. Назначение документа

Этот файл структурирует набор идей и практических требований, зафиксированных после совещания 10.06.2026, в единый план доработки MES-demo.

Рабочее название системы: **RoboPulse**.

Документ не является финальным техническим заданием. Он нужен как единая точка фиксации требований, вопросов и границ работ перед оценкой, проектированием и реализацией.

## 2. Краткое резюме требований

Основные направления доработки:

1. Расширить и улучшить редактор техпроцесса.
2. Добавить группировку изделий и/или множественный выбор изделий.
3. Добавить возможность запускать партию изделий.
4. Добавить пароли пользователям.
5. Добавить множественный выбор штук для операций лазера, зачистки и пробивного станка.
6. Убрать лишнюю третью зачистку из маршрута/техпроцесса.
7. Переименовать систему в RoboPulse.
8. Улучшить блок управления и отображение операций по печам/линиям.
9. Добавить клиентский фильтр операций в терминале участка.
10. Уточнить и включить в маршрут/операционную модель пробивной станок.

## 4. Термины и рабочие определения

### Изделие

Единица продукции, которая проходит через техпроцесс и операции: лазер, зачистка, печь, пробивной станок и другие этапы.

### Штука / unit

Конкретный экземпляр изделия внутри заказа или партии. Для MES-сценария важно, чтобы операции могли выполняться как по одной штуке, так и по нескольким выбранным штукам.

### Партия

Группа изделий/штук, запускаемая в производство одним действием. Партия должна иметь общий идентификатор, количество и набор операций, но при этом сохранять возможность отслеживать выполнение по отдельным штукам.

### Техпроцесс

Маршрут изготовления изделия: последовательность операций, участков, станков, зависимостей и нормативов.

### Клиентский фильтр

Фильтр, который настраивается пользователем на конкретном рабочем месте/терминале и влияет на отображение операций только для этого клиента/браузера/пользователя, а не на общую серверную очередь.

## 5. Эпик 1. Редактор техпроцесса

### Цель

Сделать редактор техпроцесса более удобным для настройки маршрутов изготовления, операций, станков и зависимостей между операциями.

### Требования

1. В редакторе техпроцесса должна быть возможность просматривать маршрут изделия целиком.
2. Должна быть возможность добавлять, удалять и менять порядок операций.
3. Должна быть возможность редактировать параметры операции:
   - название операции;
   - участок/рабочий центр;
   - станок или тип оборудования;
   - нормативное время;
   - обязательность операции;
   - зависимость от предыдущих операций;
   - возможность групповой обработки нескольких штук.
4. Должна быть возможность явно видеть операции для лазера, зачистки, печи и пробивного станка.
5. Нужно убрать из текущего маршрута лишнюю третью зачистку, если она действительно не требуется в производственном процессе.
6. При сохранении техпроцесса система должна валидировать маршрут:
   - маршрут не пустой;
   - нет дублей лишних операций, если они не разрешены;
   - нет циклических зависимостей;
   - указаны обязательные поля операций;
   - операции с групповым выполнением корректно помечены.

### Критерии приемки

1. Пользователь может открыть техпроцесс изделия и увидеть полный список операций.
2. Пользователь может изменить маршрут и сохранить изменения.
3. Лишняя третья зачистка отсутствует в стандартном маршруте.
4. Пробивной станок представлен как отдельная операция или рабочий центр, если он входит в реальный процесс.
5. Ошибочный техпроцесс не сохраняется без понятного сообщения об ошибке.

### Вопросы на уточнение

1. Редактор должен быть доступен только администратору/технологу или также диспетчеру?
2. Нужно ли хранить версии техпроцессов?
3. Что делать с уже запущенными партиями, если техпроцесс был изменен после запуска?
4. Третья зачистка должна быть удалена из всех изделий или только из конкретного маршрута?

## 6. Эпик 2. Группировка изделий и множественный выбор

### Цель

Позволить пользователю работать не только с одной штукой изделия, но и с группой изделий/штук: выбрать несколько позиций, объединить их в действие или запустить как партию.

### Требования

1. В списках изделий/плана должна быть возможность выбрать несколько строк.
2. Должна быть возможность выполнить групповое действие над выбранными изделиями:
   - запустить в производство;
   - отправить на операцию;
   - назначить рабочий центр;
   - изменить приоритет;
   - сформировать партию.
3. Группировка должна быть возможна минимум по:
   - номенклатуре;
   - заказу;
   - операции;
   - участку;
   - статусу;
   - партии.
4. Система должна показывать количество выбранных штук и итоговое количество в партии.
5. При групповом действии система должна проверять совместимость выбранных изделий:
   - одинаковый или совместимый техпроцесс;
   - доступность операции;
   - отсутствие конфликтов статусов;
   - отсутствие уже завершенных/заблокированных штук.

### Критерии приемки

1. Пользователь может выбрать несколько изделий в плане.
2. Пользователь видит количество выбранных штук.
3. Пользователь может запустить выбранные изделия одной партией.
4. Система не позволяет объединить несовместимые изделия без предупреждения.

### Вопросы на уточнение

1. Группировка должна быть отдельным визуальным режимом или только фильтрацией/сортировкой таблицы?
2. Можно ли объединять изделия разных заказов в одну производственную партию?
3. Какой минимальный набор групповых действий нужен в первой версии?

## 7. Эпик 3. Запуск партии

### Цель

Добавить сценарий запуска партии изделий в производство как отдельную сущность, сохраняя учет операций по каждой штуке.

### Требования

1. Пользователь должен иметь возможность создать партию из выбранных изделий/штук.
2. Партия должна содержать:
   - номер/идентификатор партии;
   - дату создания;
   - автора запуска;
   - номенклатуру или состав партии;
   - количество штук;
   - связанный заказ или список заказов;
   - текущий статус;
   - список операций;
   - список штук внутри партии.
3. После запуска партии система должна создать производственные операции для каждой штуки или корректно связать существующие unit-level операции с партией.
4. В интерфейсе должна быть видна принадлежность штуки к партии.
5. Партия должна проходить по маршруту с возможностью группового выполнения операций там, где это разрешено.
6. Должна сохраняться возможность частичного выполнения партии:
   - часть штук завершена;
   - часть находится в работе;
   - часть ожидает;
   - часть заблокирована или требует ручного решения.

### Критерии приемки

1. Пользователь может создать и запустить партию из выбранных штук.
2. Партия отображается в производственном плане.
3. Каждая штука внутри партии остается отслеживаемой отдельно.
4. Частичное выполнение партии не ломает статусы и KPI.

### Вопросы на уточнение

1. Нужен ли отдельный экран партий?
2. Может ли партия содержать разные изделия?
3. Нужно ли печатать/выгружать маршрутный лист партии?

## 8. Эпик 4. Множественный выбор штук для операций

### Цель

Сделать выполнение некоторых операций удобным для станков, где реально обрабатывается сразу несколько штук.

Приоритетные операции:

1. Лазер.
2. Зачистка на станке.
3. Пробивной станок.

### Требования

1. В терминале участка оператор должен видеть список доступных штук по выбранной операции.
2. Для операций, разрешающих групповую обработку, оператор должен иметь возможность выбрать несколько штук.
3. Для выбранных штук должны быть доступны групповые действия:
   - взять в работу;
   - поставить на паузу;
   - завершить;
   - отменить выбор;
   - передать дальше по маршруту.
4. Система должна показывать количество выбранных штук.
5. Система должна блокировать выбор штук, которые:
   - уже в работе у другого оператора;
   - находятся не на этой операции;
   - ожидают завершения предыдущей операции;
   - уже завершены;
   - имеют ошибку или требуют ручного решения.
6. После завершения групповой операции статус должен обновиться по каждой штуке отдельно.
7. История операций должна фиксировать, что действие было выполнено групповым способом.

### Критерии приемки

1. Оператор лазера может выбрать несколько штук и завершить операцию одним действием.
2. Оператор зачистки может выбрать несколько штук и завершить операцию одним действием.
3. Оператор пробивного станка может выбрать несколько штук и завершить операцию одним действием, если операция включена в маршрут.
4. Статусы всех выбранных штук обновляются корректно.
5. Нельзя случайно выполнить операцию над штуками, которые не готовы к этой операции.

### Вопросы на уточнение

1. Нужно ли указывать фактическое время обработки одно на всю группу или отдельно на каждую штуку?
2. Нужно ли ограничение максимального количества штук за одно групповое действие?
3. Нужно ли подтверждение перед массовым завершением операции?

## 9. Эпик 5. Пользователи и пароли

### Цель

Добавить базовую авторизацию пользователей через пароль, чтобы доступ к системе и действиям были персональными.

### Требования

1. У пользователя должен быть пароль.
2. Пароли должны храниться только в хешированном виде.
3. Должен появиться вход в систему по логину и паролю.
4. Сессия пользователя должна сохраняться между обновлениями страницы в рамках допустимого времени жизни.
5. Должна быть возможность выйти из системы.
6. Должны быть роли или права доступа минимум для следующих типов пользователей:
   - администратор;
   - технолог;
   - диспетчер;
   - оператор участка;
   - руководитель/директор.
7. Действия в системе должны фиксировать пользователя, который их выполнил:
   - запуск партии;
   - изменение техпроцесса;
   - старт операции;
   - завершение операции;
   - изменение фильтров, если фильтр сохраняется на сервере;
   - административные действия.

### Критерии приемки

1. Нельзя войти в систему без корректного пароля.
2. Пароль не хранится в открытом виде.
3. Пользователь может выйти из системы.
4. Оператор не видит административные функции, если у него нет прав.
5. В журнале действий видно, кто выполнил ключевое действие.

### Вопросы на уточнение

1. Нужна ли самостоятельная смена пароля пользователем?
2. Нужен ли сброс пароля администратором?
3. Нужна ли интеграция с внешней учетной системой или достаточно локальных пользователей?

## 11. Эпик 7. Блок управления и панели операций

### Цель

Уточнить и улучшить блок управления операциями, особенно отображение нижней/верхней панели и распределение операций по печам.

Исходная формулировка:

```text
Блок управления нижняя панель, верхняя панель вторая из 10.
На одну печь ряд операций одна печь ряд от операций.
```

Формулировка требует уточнения, но предварительно ее можно разложить так:

1. В интерфейсе есть блок управления с верхней и нижней панелью.
2. Нужно изменить расположение или состав элементов на нижней панели.
3. Нужно доработать верхнюю панель, возможно отображение "2 из 10" или выбор второй операции из десяти.
4. Для печей требуется группировать/показывать ряд операций по конкретной печи.
5. Возможно, нужна схема: одна печь = свой ряд операций.

### Предварительные требования

1. Для каждой печи должен быть виден список операций, относящихся к ней.
2. Если печей несколько, операции должны быть визуально разделены по печам.
3. Пользователь должен понимать:
   - какая печь выбрана;
   - какие операции ожидают;
   - какие операции в работе;
   - какие операции завершены;
   - сколько операций всего и какая сейчас активная.
4. В блоке управления не должно быть лишних кнопок, не относящихся к выбранной операции/печи.
5. Панели должны быть удобны для терминала участка.

### Критерии приемки

1. Оператор видит операции, сгруппированные по печи или рабочему центру.
2. Интерфейс показывает текущую позицию операции в очереди/маршруте, например "2 из 10", если это подтвержденный сценарий.
3. Нижняя и верхняя панели не дублируют друг друга и не мешают работе оператора.

### Вопросы на уточнение

1. Что именно означает "верхняя панель вторая из 10": номер операции, номер вкладки, счетчик изделий или счетчик операций?
2. Нужно ли показывать операции печи в виде горизонтального ряда, таблицы или карточек?
3. Одна печь может выполнять несколько разных операций или только одну операцию с разными партиями?
4. Нужно ли учитывать загрузку печи по времени?

## 12. Эпик 8. Клиентский фильтр операций в терминале участка

### Цель

Позволить оператору на терминале участка настроить собственный фильтр операций, чтобы видеть только релевантную очередь.

### Требования

1. В терминале участка должен быть фильтр операций.
2. Фильтр должен настраиваться пользователем на конкретном клиенте/рабочем месте.
3. Фильтр не должен менять глобальную очередь операций для других пользователей.
4. Фильтр должен позволять выбрать:
   - участок;
   - операцию;
   - станок;
   - статус;
   - партию;
   - заказ;
   - номенклатуру.
5. Фильтр должен сохраняться после обновления страницы.
6. Пользователь должен иметь возможность быстро сбросить фильтр.
7. Если фильтр скрывает все операции, интерфейс должен явно показать, что список пуст из-за фильтра.

### Варианты хранения

#### Вариант A. Локальное хранение

Фильтр хранится в браузере через localStorage.

Плюсы:

1. Быстро реализовать.
2. Не требуется менять backend.
3. Подходит для сценария "фильтр конкретного терминала".

Минусы:

1. Фильтр не переносится на другой браузер или устройство.
2. Очистка браузера удалит настройки.

#### Вариант B. Серверное хранение

Фильтр хранится на backend как настройка пользователя или терминала.

Плюсы:

1. Настройки можно восстановить после входа.
2. Можно управлять фильтрами централизованно.
3. Можно аудировать изменения.

Минусы:

1. Требуется модель данных и API.
2. Нужно решить, фильтр привязан к пользователю или к терминалу.

### Рекомендация для первой версии

Начать с локального клиентского фильтра в браузере. Если после внедрения потребуется переносимость настроек между терминалами, добавить серверное хранение.

### Критерии приемки

1. Оператор может выбрать операцию и видеть только подходящие задания.
2. Фильтр сохраняется после обновления страницы.
3. Фильтр можно сбросить.
4. Фильтр не влияет на другие терминалы и пользователей.

## 13. Эпик 9. Пробивной станок

### Цель

Включить пробивной станок в производственную модель и интерфейс, если он является отдельным этапом маршрута.

### Требования

1. Пробивной станок должен быть доступен как рабочий центр/станок.
2. В техпроцессе должна быть операция пробивки, если она применяется к изделию.
3. Операции пробивного станка должны отображаться в терминале участка.
4. Для пробивного станка должна быть возможность множественного выбора штук, если реальный процесс это допускает.
5. Операция пробивки должна участвовать в зависимостях маршрута:
   - не стартует раньше предыдущей обязательной операции;
   - после завершения открывает следующую операцию;
   - влияет на прогресс партии/изделия.

### Критерии приемки

1. Пробивной станок виден в списке рабочих центров.
2. Операция пробивки может быть добавлена в техпроцесс.
3. Оператор пробивного станка видит свои задания в терминале.
4. По нескольким выбранным штукам можно выполнить групповое действие, если оно разрешено для этой операции.

### Вопросы на уточнение

1. Пробивной станок всегда идет после лазера или может быть в другой позиции маршрута?
2. Пробивка нужна для всех изделий или только для части номенклатуры?
3. Нужны ли отдельные параметры операции: программа, инструмент, материал, толщина?

## 16. Риски и ограничения

1. Запуск партий может затронуть существующую unit-level модель выполнения операций.
2. Множественное завершение операций требует аккуратной проверки статусов, чтобы не завершать неготовые штуки.
3. Пароли и роли требуют аккуратного хранения учетных данных и миграции существующих пользователей.
4. Редактор техпроцесса может потребовать изменения структуры данных, если текущая модель маршрута слишком статична.
5. Блок управления печами пока описан неоднозначно и требует отдельного UX-уточнения.
6. Если существующие данные уже содержат запущенные операции, удаление третьей зачистки нужно делать аккуратно: отдельно для шаблонов маршрутов и отдельно для уже запущенных производственных единиц.
